# Архитектурная документация проекта MyVCS

## 1. Введение

### 1.1 Цель документа
Данный документ описывает архитектуру учебного проекта MyVCS — упрощённой системы контроля версий, вдохновлённой Git. Документ предназначен для членов команды и преподавателей.

### 1.2 Область применения
MyVCS представляет собой консольное приложение для управления версиями файлов, поддерживающее базовые операции: инициализацию репозитория, добавление файлов, коммиты, просмотр истории и переключение между ревизиями.

### 1.3 Термины и определения

| Термин | Определение |
|--------|-------------|
| **Blob** (Binary Large Object) | Объект хранения содержимого файла |
| **Tree** | Объект, представляющий директорию с ссылками на blobs и другие trees |
| **Commit** | Снимок состояния репозитория в определённый момент времени |
| **Index** (Staging Area) | Промежуточная область для подготовки файлов к коммиту |
| **HEAD** | Указатель на текущую ветку или коммит |
| **Revision** | Конкретное состояние репозитория (коммит или ветка) |

## 2. Формализованные требования

### 2.1 Функциональные требования

#### FR-001: Инициализация репозитория
- Система должна создавать структуру директорий `.myvcs/` с поддиректориями `objects/` и `refs/heads/`
- Система должна создавать файл `HEAD`, указывающий на ветку `master` по умолчанию
- При попытке повторной инициализации система должна выводить сообщение о существовании репозитория

#### FR-002: Добавление файлов в индекс
- Система должна поддерживать добавление отдельных файлов и директорий
- Система должна игнорировать служебную директорию `.myvcs`
- Система должна вычислять SHA-256 хеш содержимого файла
- Система должна сохранять файлы как blob-объекты в базу объектов с zlib-компрессией

#### FR-003: Создание коммита
- Система должна строить иерархическое дерево (Tree) из плоского индекса
- Система должна создавать commit-объект с метаданными (автор, время, сообщение)
- Система должна обновлять указатель текущей ветки
- Система должна очищать индекс после успешного коммита

#### FR-004: Просмотр истории
- Система должна выводить историю коммитов от текущего HEAD до корневого коммита
- Формат вывода должен включать: хеш, автора, дату, сообщение

#### FR-005: Переключение между ревизиями
- Система должна поддерживать переключение по имени ветки или хешу коммита
- Система должна восстанавливать файлы из указанной ревизии
- Система должна обновлять индекс и HEAD

### 2.2 Нефункциональные требования

#### NFR-001: Производительность
- Время добавления файла должно быть O(n), где n — размер файла
- Время коммита должно быть O(m log m), где m — количество файлов в индексе

#### NFR-002: Надёжность
- Система должна использовать атомарные операции записи файлов
- Система должна валидировать целостность объектов при чтении

#### NFR-003: Переносимость
- Код должен работать на Windows, Linux, macOS
- Пути файлов должны нормализоваться для кроссплатформенности

#### NFR-004: Расширяемость
- Архитектура должна позволять добавление новых команд без изменения ядра
- Архитектура должна позволять замену хранилища объектов

### 4.1 Описание ключевых классов

#### Model Layer

| Класс | Описание |
|-------|----------|
| **VcsObject** | Интерфейс, определяющий контракт для всех объектов VCS. Методы `type()` и `serialize()` обеспечивают полиморфное поведение. |
| **Blob** | Хранит содержимое файла как массив байтов. Не содержит метаданных (имя файла хранится в Tree). |
| **Tree** | Представляет директорию. Содержит список `TreeEntry` с режимом доступа, типом, хешем и именем. |
| **TreeEntry** | Запись о файле или поддиректории в дереве. Использует tab-разделитель для совместимости с Git-форматом. |
| **Commit** | Снимок состояния репозитория. Содержит ссылку на корневое дерево, родительские коммиты, автора, время и сообщение. |
| **CommitNode** | Вспомогательный record для связывания хеша коммита с его содержимым (используется в `log`). |

#### Storage Layer

| Класс | Описание |
|-------|----------|
| **ObjectStorage** | Ключевой класс персистентности. Сохраняет объекты с zlib-компрессией, загружает и распаковывает их. Формат пути: `.myvcs/objects/xx/yyyy...` |
| **HashUtils** | Утилитный класс для вычисления SHA-256 хешей. Использует `MessageDigest` из стандартной библиотеки Java. |

#### Core Layer

| Класс | Описание |
|-------|----------|
| **VcsService** | Фасад для операций VCS. Координирует работу `IndexManager`, `ReferenceManager` и `ObjectStorage`. Содержит логику построения дерева из плоского индекса. |
| **IndexManager** | Управляет staging area. Хранит соответствие путей и хешей в `TreeMap` для упорядочивания. Формат индекса: `<hash> <path>` на каждой строке. |
| **ReferenceManager** | Управляет ссылками: HEAD и ветки. Поддерживает разрешение имён веток в хеши коммитов. Обрабатывает detached HEAD. |
| **RepositoryManager** | Создаёт начальную структуру директорий репозитория. Статический класс без состояния. |
| **ObjectParser** | Фабрика для десериализации объектов. Маршрутизирует парсинг по типу объекта в заголовке. |

#### CLI Layer

| Класс | Описание |
|-------|----------|
| **CommandParser** | Парсер командной строки. Использует паттерн Command: хранит реестр команд в `Map<String, Command>`. |
| **Command** | Интерфейс команды. Позволяет добавлять новые команды без изменения парсера (OCP). |
| **InitCommand** | Инициализирует репозиторий. Не требует зависимостей. |
| **AddCommand** | Добавляет файлы в индекс. Зависит от `IndexManager`. |
| **CommitCommand** | Создаёт коммит. Зависит от `VcsService`. Парсит флаг `-m` для сообщения. |
| **LogCommand** | Выводит историю. Зависит от `VcsService`. |
| **CheckoutCommand** | Переключает ревизию. Зависит от `VcsService`. |

## 5. Обоснование выбора технологий

### 5.1 Язык программирования: Java 17+

**Выбор:** Java 17 (LTS)

**Обоснование:**

| Критерий | Обоснование |
|----------|-------------|
| **Кроссплатформенность** | Java обеспечивает переносимость между Windows, Linux, macOS без изменений кода (требование NFR-003). `Path` API корректно обрабатывает разделители путей. |
| **Богатая стандартная библиотека** | `java.nio.file` предоставляет мощный API для работы с файловой системой. `java.security.MessageDigest` — для SHA-256. `java.util.zip` — для zlib-компрессии. Внешние зависимости не требуются. |
| **Современные возможности** | Java 17+ поддерживает records (для immutable DTO), pattern matching для `instanceof`, text blocks. Это упрощает код модели. |

### 5.4 Архитектурные решения

#### 5.4.1 Слоистая архитектура (Layered Architecture)

**Решение:** Разделение на 4 слоя: CLI → Core → Model → Storage

**Обоснование:**
- **Separation of Concerns:** Каждый слой имеет единственную ответственность.
- **Тестируемость:** Слои можно тестировать изолированно через mock-объекты.
- **Замена реализаций:** Storage Layer можно заменить на базу данных без изменения Core.

#### 5.4.2 Паттерн Command

**Решение:** Интерфейс `Command` с реализациями для каждой операции

**Обоснование:**
- **Open/Closed Principle:** Новые команды добавляются без изменения `CommandParser`.
- **Single Responsibility:** Каждая команда инкапсулирует логику одной операции.
- **Тестируемость:** Команды можно тестировать независимо.

#### 5.4.3 Формат хранения объектов (Git-compatible)

**Решение:** Формат `<type> <size>\0<content>` с zlib-компрессией

**Обоснование:**
- **Проверенное решение:** Формат Git доказал свою эффективность на протяжении 20 лет.
- **Дедупликация:** SHA-256 хеширование автоматически исключает дубликаты файлов.
- **Простота отладки:** Объекты можно исследовать стандартными утилитами (zlib, hexdump).

#### 5.4.4 Отказ от внешних библиотек

**Решение:** Использование только стандартной библиотеки Java

**Обоснование:**
- **Учебный характер:** Цель проекта — понимание внутреннего устройства VCS, а не использование готовых решений.
- **Минимизация зависимостей:** Уменьшает сложность проекта и риск конфликтов версий.
- **Портативность:** Проект можно собрать и запустить на любой системе с JDK 17+ без скачивания дополнительных библиотек.
